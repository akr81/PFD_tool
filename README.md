# PFD Sketch

ブラウザ上で PFD（Process Flow Diagram）、S&T（Strategy and Tactics Tree）、CRT（Current Reality Tree）、EC（Evaporating Cloud）、CCPM（Critical Chain Project Management）、Goal Navi（フェーズ単位の進捗・バッファ管理）をすばやく扱うためのシングルファイルツールです。PFDでは成果物とプロセスを交互に接続しながら、工程の見通し、担当者、見積もり、進捗、クリティカルパスを軽く管理できます。

## 起動方法

`open-pfd.cmd` をダブルクリックすると、`index.html` がブラウザで開きます。

直接 `index.html` をブラウザで開いても使えます。データはブラウザのローカルストレージに自動保存されます。

## 基本概念

画面上部のセレクトボックスで、図の種類を切り替えられます。

### PFD

PFDでは、次の2種類のエンティティを扱います。

- 成果物: 四角形のノード
- プロセス: 楕円形のノード

成果物どうし、プロセスどうしは接続できません。接続候補として不適切な操作はグレーアウトされます。

### S&T

S&Tでは、1つの項目に次の5つの欄を持たせます。

- 必要条件仮定
- 戦略
- 並行仮定
- 戦術
- 充分条件仮定

ノード上の `+` は参照先となる上位のS&T項目、下の `+` は参照元となる下位のS&T項目を追加します。左/右の `+` は、参照先がある場合に同じ上位へ向かう同階層項目を追加します。矢印は下位項目から上位項目へ、下から上に向かって描かれます。

### CRT

CRTでは、現状の症状・結果と、その原因となる状況を縦方向につなぎます。下側が原因、上側が症状・結果になるように、矢印は下から上へ向かって描かれます。

ノード上の上 `+` は結果を追加し、下 `+` は原因を追加します。左/右の `+` は、結果がある場合に同じ結果へ向かう原因を追加します。

各エンティティは色を持てます。詳細パネルから、なし・青・黄・赤・緑・橙・紫・灰を選べます。

CRTのエッジを選択するとAND条件を設定できます。同じAND番号を付けた複数の原因は、小さなANDノードで合流してから結果へ接続されます。

### CCPM

CCPMでは、タスクを左から右へ依存関係でつなぎ、見積もり期間と担当者を使って簡易スケジュールを計算します。

- 右の `+`: 後続タスクを追加
- 左の `+`: 先行タスクを追加
- 上/下の `+`: 後続タスクがある場合、その後続への並列先行タスクを追加
- タスクは楕円、手動で追加するバッファは四角で表示します

担当者が同じタスクは同時に走れないものとして扱い、依存関係と担当者制約を合わせてクリティカルチェーンを判定します。CCPMモードでは画面下部にガントチャートが表示され、依存関係線、タスクバー、クリティカルチェーン、フィーディングバッファ、プロジェクトバッファを確認できます。

上部のプロジェクト設定ボタンから、開始日、終了日、今日、休日、リソース枠、ベースラインを設定できます。設定内容はJSON保存/読込と自動保存に含まれます。リソース枠は担当者入力の候補として使われ、枠数は全体の同時実行上限としても扱われます。`Akira:2` のように書くと担当者別の同時実行容量を増やせます。

ガントチャート上端のつまみを上下にドラッグすると、下部の表示領域の高さを変更できます。変更した高さは自動保存とJSON保存に含まれます。画面上部の集計、CCPM Gantt、ガント下部のフィーバーチャートで、クリティカルチェーン進捗率とプロジェクトバッファ消費率を確認できます。フィーバーチャートは今日時点の履歴を記録し、折れ線として表示できます。

### EC

ECでは、固定構造の5要素と4つの仮定欄を扱います。

- A: 共通目的
- B/C: 目的・要望
- D/D': 対立する行動
- B→A、C→A、D→B、D'→C: 各矢印の仮定

ECの構造は固定なので、ノード追加・エッジ削除は行いません。中央クリックで各欄をすぐ編集でき、D と D' の対立関係は赤いイナズマ線で表示されます。

### Goal Navi

Goal Naviは、まだタスク分解しきれていない段階でも、フェーズ単位で進捗とバッファ消費を追うための画面です。ノードやエッジを描く図ではなく、プロジェクト設定、フェーズ、進捗記録、フィーバーチャートをまとめて扱います。

- プロジェクト設定: 開始日、終了日、休日、目標完了日までの割合を設定します
- フェーズ: 任意の数だけ追加でき、各フェーズに計画日数を置きます
- 進捗記録: 記録日ごとに、残計画日数、合計見積、完了人日、メモを入力します
- チームベロシティ: 前回記録からの完了人日の増分を稼働日数で割って計算します
- 仮置き速度: ベロシティが0以下、計算不能、または一時的に採用したくない場合に手入力できます
- フィーバーチャート: 進捗率とバッファ消費率をプロットし、安全・注意・危険・遅延リスク大を判定します

進捗記録は最新のものが上に表示されます。メモに「チャート表示」を付けると、フィーバーチャート上の点の近くにメモが表示されます。仮置き速度を使った点は、通常の丸ではなく菱形で表示されます。

Goal Naviの同梱サンプルは、2026/5/1開始、2026/9/30完了、3フェーズ、目標完了日までの割合66.7%のプロジェクトを、週1回評価した終盤シナリオです。序盤は安全・注意を往復し、中盤は手戻りで危険域に入り、終盤はバッファを回復しながら完了目前まで進む流れを確認できます。

## 主な操作

- ノード中央付近をクリック: タイトル編集
- ノード周辺をクリック: 選択
- ノードをダブルクリック: 右側の詳細パネルを表示
- ノード右の `+`: 接続先を追加
- ノード左の `+`: 接続元を追加
- ノード上/下の `+`: 接続先がある場合、その接続先への並列入力を追加
- エッジ中央のハンドルをクリック: エッジ選択、選択中に再クリックで削除
- ラベルや集計値にマウスオーバー: 用語や計算の考え方をツールチップで表示

ECでは `+` やエッジ削除は使わず、各カードの内容編集を中心に使います。

Goal Naviではノード操作は使わず、プロジェクト設定、フェーズ、進捗記録の入力欄を直接編集します。

## キーボード操作

- 矢印キー: 選択ノードを切り替え
- `F2`: タイトル編集
- `Enter`: 下方向に追加
- `Shift + Enter`: 上方向に追加
- `Tab`: 右方向に追加
- `Shift + Tab`: 左方向に追加
- `Delete`: 選択中のノードまたはエッジを削除
- `Ctrl + Z`: Undo
- `Ctrl + Y` / `Ctrl + Shift + Z`: Redo

## 表示モード

### シンプルモード

タイトルと接続関係を中心に、すっきり表示します。クリティカルパスの赤線表示は行いません。

### 詳細モード

ノード上に次の情報を表示します。

- 左上: 状況アイコン
- 左下: 見積もり期間
- 右下: 担当者

詳細モードでは、左上・左下・右下をクリックしてノード上から直接編集できます。担当者は、既に入力済みの担当者名を候補としてプルダウン表示できます。

上部には次の集計も表示されます。

- 全プロセスの見積もり期間合計
- 完了アイコン付きプロセスの見積もり期間合計
- 全体進捗率
- クリティカルパス長
- クリティカルパス上の完了分
- クリティカルパス上の進捗率

文字サイズスライダーで、詳細モード内のタイトル・メタ情報・アイコンの表示サイズを比率を保ったまま調整できます。

S&Tでは詳細モードにすると、ノード上に5つの欄をまとめて表示します。

CRTでは、詳細パネルで課題・状況の本文と色を編集できます。

CCPMでは、詳細モードでクリティカルチェーン上のタスクとエッジを強調表示します。下部のガントチャートはシンプル/詳細のどちらでも表示されます。

ECでは、シンプルモードではA/B/C/D/D'だけを表示し、詳細モードで各矢印の仮定カードを表示します。PFD用の進捗パネルやクリティカルパス表示は出ません。

## 詳細パネル

ノードをダブルクリックすると右側に詳細パネルが開きます。

詳細パネルでは、次の情報を編集できます。

- タイトル
- 見積もり期間
- 担当者
- 状況
- Markdown形式の本文

表示中の詳細をクリックすると編集状態になります。Markdown本文だけは編集時にテキストエリアへ切り替わり、それ以外は表示時に近い見た目のまま編集できます。

S&T項目の場合は、詳細パネルで5つの欄を直接編集できます。CRTの場合は課題・状況と色を編集できます。ECの場合は、選択したカードの問いと内容を編集できます。
CCPMタスクの場合は、タイトル、見積もり期間、担当者、状況、開始日、終了日、残日数、完了、Markdown本文に加えて、計算された開始・終了とクリティカルチェーン判定を確認できます。CCPMバッファの場合は、担当者や状況を持たず、見積もり期間を持つ四角いノードとして扱います。

## 見積もり期間

見積もり期間は、プロセスの集計とクリティカルパス判定に使われます。

単位なしの数値は「日」として扱います。主な単位の扱いは次の通りです。

- `日`, `d`, `day`: 日
- `時間`, `h`, `hour`: 8時間 = 1日換算
- `分`, `min`: 8時間 = 1日換算
- `週`, `w`, `week`: 5日換算
- `月`, `month`: 20日換算

例: `3`, `3日`, `16h`, `2週`

## クリティカルパス

詳細モードでは、プロセスの見積もり期間を重みとして、接続方向に沿った最長経路をクリティカルパスとして判定します。

- 成果物の期間は 0 として扱います
- プロセスの見積もり期間だけを加算します
- クリティカルパス上のエッジは赤く表示されます
- シンプルモードでは赤線表示を行いません

## CCPMスケジュール

CCPMモードでは、タスクの見積もり期間、開始日、終了日、残日数、完了状態を使って開始日・終了日を計算します。

- 依存関係は左から右へ流れるタスク順として扱います
- 同じ担当者のタスクが重なる場合は、後続パスが長い側を先にする仮想の担当者制約を追加して再計算します
- 依存関係と担当者制約を合わせた最長の制約列をクリティカルチェーンとして表示します
- 詳細モードでは、依存関係だけで見たクリティカルパスを青点線、担当者制約込みのクリティカルチェーンを赤線で表示します
- クリティカルチェーン上の担当者制約は、点線の赤い補助矢印で表示します
- プロジェクト設定の開始日、休日、今日を使って、ガント上のD日表示を実日付に変換します
- 開始日が入っている未完了タスクは、見積もり期間ではなく残日数を残作業として扱います
- CCPMタスクに進行中ステータスを付けると、開始日が未設定ならプロジェクト設定の「今日」を開始日に入れます。完了ステータスを付けると、終了日が未設定なら「今日」を終了日に入れ、残日数を0にします
- ガントチャート上部の「進捗更新」から、今日の日付、各タスクの未着手/進行中/完了、残日数、開始日、終了日、トラブル有無を一覧で更新できます。保存時にフィーバーチャート履歴も同時に記録できます
- ガントチャート上部の「チェック」から、見積もり不備、担当者未設定、残日数未更新、終了日未設定、PB/FB消費、ベースライン完了予定日と `BL CC + PB` の不一致、依存ループ、CC上の未更新タスクなどを一覧で確認できます。タスクに紐づく項目は一覧から選択できます
- 完了タスクは残作業0日として扱い、完了分の進捗計算には元の見積もり期間を使います
- 開始日と終了日を設定してベースライン登録すると、その時点のクリティカルチェーン長とプロジェクトバッファを固定します
- ベースライン登録時にはタスクごとの開始/終了位置も保存します。ガント上には薄いベースラインバーが表示され、タスクごとの現在計画との差分を `計画差` として確認できます。古いJSONのようにタスク別ベースラインがない場合は、現在の計画条件から推定したBLバーを点線で表示します
- フィーバーチャートでは、クリティカルチェーン上の完了・残日数から進捗率を出し、今日時点のバッファ消費率と残見込み期間を表示します。ガント上部の `完了差` は、現在の見込み完了日がベースライン完了予定日から何日ずれているかを表します。ベースライン完了予定日は登録済みの完了予定日を優先し、古いJSONなどで未登録の場合のみ `BL CC + PB` から推定します。バッファ消費率は100%超過後も上限を設けず表示し、チャートの縦軸は必要に応じて広がります。履歴記録ボタンでその日の点を保存でき、チャート右側の履歴一覧から選択・強調表示・削除・メモ入力ができます
- 依存関係にループがある場合は、該当タスク・エッジ・ガント線を警告色で表示します
- 見積もり期間が未入力または読取不可の場合は、警告を出して `1日` として計算します
- プロジェクトバッファは、ベースライン登録済みなら登録時のクリティカルチェーン終了位置とバッファ長で固定表示し、未登録で開始日・終了日がある場合はプロジェクト稼働日数とクリティカルチェーン長の差分として表示します
- フィーディングバッファは、非クリティカル側からクリティカル側へ合流する依存関係の手前に表示します。各FBには消費率を表示し、75%以上で注意、100%以上で警告色になります
- 担当者は `Akira, Mika` のように複数指定でき、いずれかの担当者容量を使うタスクとしてリソース競合を判定します

これはローカルで素早く使うためのCCPM計算です。休日を除いた稼働日、担当者制約、担当者ごとの容量、リソース枠による同時実行上限を扱います。時間帯単位の稼働率はまだ扱いません。

## Goal Naviの計算

Goal Naviでは、タスクではなくフェーズ単位の概算値から、プロジェクト全体の見通しを計算します。

- 総稼働日数: プロジェクト開始日から終了日までのうち、土日祝日・休日を除いた日数
- 目標完了日: 総稼働日数に「目標完了日までの割合」を掛けた地点の日付
- バッファ日数: 目標完了日からプロジェクト終了日までの余裕日数
- 残計画日数: まだ見積もり済みタスクに分解されていない、フェーズ単位の粗い残日数
- 合計見積: 見積もり済みタスクの合計人日
- 完了: 完了した見積もり済みタスクの見積人日合計
- 見積残: 合計見積から完了を引いた値
- チームベロシティ: 前回記録から増えた完了人日を、前回からの稼働日数で割った値
- 推定完了日: 記録日に、残計画日数と見積残をチームベロシティで割った残日数を足して算出した完了見込み日
- バッファ消費率: 推定完了日が、目標完了日からプロジェクト終了日までのバッファをどの程度消費しているか

フィーバーチャートの判定ラインは次の通りです。

- 安全: `バッファ消費率 <= 0.6 * 進捗率 + 10`
- 注意: `バッファ消費率 <= 0.6 * 進捗率 + 25`
- 危険: 注意ラインを超え、100%以下
- 遅延リスク大: 100%超過

チームベロシティが0以下または計算不能の場合は赤太字で表示され、推定完了日は計算不能になります。前回から30%以上変動した場合は黄色で表示されます。どの記録でも仮置き速度を入力でき、入力した場合は推定完了日の計算に優先して使われます。

## 保存と読み込み

ブラウザのローカルストレージに自動保存されます。

上部のファイル操作から、JSONとして保存・読み込みもできます。JSONにはノード、エッジ、詳細情報、表示設定が含まれます。

ファイル操作の「サンプル」プルダウンから、同梱サンプルを直接読み込めます。サンプルとして `sample/sample_curry.json`、`sample/sample_stt.json`、`sample/sample_crt.json`、`sample/sample_crt_study_time.json`、`sample/sample_ec.json`、`sample/sample_ccpm.json` を同梱しています。CCPMの元ツール相当の確認用に、序盤・中盤・終盤の3状態を持つ `sample/sample_ccpm_original_like*.json` もあります。担当者制約確認用に `sample/sample_ccpm_resource_priority.json`、警告確認用に `sample/sample_ccpm_warnings.json` もあります。Goal Naviの週次評価サンプルとして `sample/sample_goal_navi_late_scenario.json` もあります。

## ローカル利用とセキュリティ

このツールは完全ローカルで動作する想定です。通常の使い方ではサーバーや外部サービスは不要で、外部通信も行いません。

注意点は次の通りです。

- 自動保存データはブラウザのローカルストレージに残ります
- JSON保存ファイルは平文なので、機密情報を含む場合は保存場所のアクセス権に注意してください
- Markdown本文のリンクをクリックすると、指定された外部サイトをブラウザで開きます
- 簡易Webサーバーで開く場合は、LANに公開されないよう `127.0.0.1` で起動してください
- 共有PCやブラウザ拡張機能が多い環境では、ローカルストレージや画面内容を読まれる可能性があります

機密案件で使う場合は、作業後にブラウザのサイトデータを削除するか、JSON保存先を暗号化・アクセス制限されたフォルダに置く運用を推奨します。
